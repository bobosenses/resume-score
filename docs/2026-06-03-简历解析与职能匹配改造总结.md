# 简历解析与职能匹配改造总结

> 日期：2026-06-03
> 状态：已完成并上线
> 关键词：textSpan模式、LLM辅助匹配、性能优化

---

## 一、背景

### 1.1 简历解析的痛点

原有方案将简历全文 + 解析规则送入 LLM，要求 LLM 输出完整的结构化 JSON，其中 `description` 字段需要 LLM 逐字粘贴简历原文。

**问题**：
- 单次解析输出 212KB（约 11.8 万 tokens），耗时 80 秒
- 5 并发时最慢请求 107 秒
- LLM 粘贴原文可能截断、遗漏或篡改个别字词

### 1.2 职能匹配的痛点

原有方案使用 Embedding + Rerank 纯向量搜索，从 1567 条职能标签中匹配最相关的 5 条。

**问题**：
- 精确匹配率仅 22%，含二级匹配也只有 25%
- 向量搜索无法区分「硬件工程师」和「软件工程师」等细微差别
- 长简历原文直接搜索会稀释语义，导致匹配跑偏

---

## 二、简历解析改造：textSpan 模式

### 2.1 核心思路

将 LLM 的任务从「理解语义 + 搬运文本」简化为「只做语义理解」。`description` 字段的原文截取交给后端代码完成。

### 2.2 架构

```
第一步：LLM 输出骨架（~20s）
  简历原文 + 解析规则 → LLM → 精简JSON（textSpan替代description）

第二步：后端代码拼装（<100ms）
  骨架JSON + 简历原文 → /api/assemble → 完整JSON（含description）
```

### 2.3 textSpan 规则

经历对象的字段从：

```json
{
  "companyName": "某乘用车集团",
  "position": "品牌院软件总工",
  "duration": "2025.08-至今",
  "description": "主责电子电器产品和软件的开发和项目管理工作：1、作为电子电器中心的CPL...（整段原文）"
}
```

改为：

```json
{
  "companyName": "某乘用车集团",
  "position": "品牌院软件总工",
  "duration": "2025.08-至今",
  "textSpan": "某乘用车集团（2025.08 - 至今"
}
```

`textSpan` 是简历原文中该段经历的起始锚点文本（约 10-30 字），后端用字符串匹配从原文中截取完整 description。

### 2.4 拼装逻辑

```javascript
function assembleDescriptions(experiences) {
    for (let i = 0; i < experiences.length; i++) {
        const span = experiences[i].textSpan;
        const start = resume.indexOf(span);          // 定位起始位置
        const nextSpan = experiences[i + 1]?.textSpan;
        const end = nextSpan ? resume.indexOf(nextSpan) : resume.length;  // 定位结束边界
        experiences[i].description = resume.substring(start, end).trim(); // 截取原文
        delete experiences[i].textSpan;
    }
}
```

### 2.5 关键改动

| 文件 | 改动 |
|------|------|
| `config/parse_rules.json` | v3.0：经历对象用 textSpan 替代 description |
| `web/server.js` | 新增 `POST /api/assemble` 接口 |
| `web/index.html` | 流式输出结束后自动调用 assemble |

### 2.6 效果

| 指标 | 改造前 | 改造后 | 变化 |
|------|--------|--------|------|
| LLM 输出量 | 212KB（11.8万 tokens） | 74KB（4.1万 tokens） | **-65%** |
| 单请求耗时 | ~80s | ~20s | **-75%** |
| description 精度 | LLM 粘贴（可能截断） | 原文截取（100% 精确） | **更高** |

### 2.7 额外优化

| 优化项 | 措施 | 效果 |
|--------|------|------|
| vLLM 并发 | 添加 `--enable-chunked-prefill` + `--max-num-seqs 16` | 5并发最慢请求从107s降至83s |
| Prompt 精简 | 解析规则从 16KB 压缩到 2.2KB | prefill 阶段更快 |

## 三、职能匹配改造：LLM 辅助模式

### 3.1 第一阶段：关键词提取

简历原文太长（3000+ 字），直接搜索会稀释语义。实现 `extractSearchQuery` 函数，从简历中提取高密度关键词：

- **职位名称**：匹配「公司名（时间）职位」模式，过滤停止词（下属人数、职责业绩等）
- **求职意向**：提取目标职位
- **自我评价**：支持带标签和无标签两种格式，按句号截取短句

解决了「医院院长搜不到」的问题——从整段简历中提取「院长助理/经营院长 市场总监 运营总监 医院管理人员」作为搜索关键词。

### 3.2 第二阶段：LLM 辅助匹配（核心改造）

**架构**：

```
简历原文
  ↓
提取关键词（extractSearchQuery）
  ↓
ChromaDB 向量检索 + Rerank → Top 20 候选（2-3s）
  ↓
LLM 从候选中选 Top 5 + 匹配理由（2-3s）
  ↓
返回结果（含置信度 + 理由）
```

**LLM Prompt 设计**（控制在 ~700 tokens）：

```
你是职能匹配专家。根据简历信息，从以下候选职能中选出最匹配的Top 5。

【简历信息】
{提取的简历摘要，限600字}

【候选职能列表】
1. 电子控制器硬件-底盘电子控制器硬件-底盘控制器硬件工程师
2. 产品软件研发-底盘控制应用层软件与算法-制动控制软件工程师
...

严格按以下JSON格式输出，不要解释：
{"matches":[{"rank":1,"candidate_index":1,"confidence":"high","reason":"一句话理由"},...]}
```

**降级策略**：LLM 调用失败时自动降级为纯向量搜索结果。

### 3.3 关键改动

| 文件 | 改动 |
|------|------|
| `web/server.js` | `/api/function-search` 改造：向量召回 + LLM 判断 |
| `web/server.js` | 修复 MODEL 变量不匹配 vLLM 模型 ID |
| `web/server.js` | LLM 调用禁用 thinking 模式 |
| `web/index.html` | 结果卡片新增置信度标签和匹配理由 |
| `import_functions.py` | 清洗 666 条空壳数据（2233→1567 条） |

### 3.4 效果

| 指标 | 纯向量模式 | LLM 辅助模式 | 提升 |
|------|-----------|-------------|------|
| ✅ 精确匹配（三级全对） | 22% | **34%** | **+55%** |
| ✅+🟡 匹配率（含二级） | 25% | **44%** | **+76%** |
| ❌ 不匹配 | 44% | 37% | -7% |

**LLM 解决的关键案例**：

| 案例 | 旧模式结果 | LLM 结果 | 说明 |
|------|-----------|---------|------|

| 半导体封装工程师 | 半导体器件测试 ❌ | 功率模块封装工艺 ✅ | LLM 分清封装和测试 |
| 五官科医生 | 急诊科医生 ❌ | 五官科医生 ✅ | LLM 区分不同医学专科 |

## 四、部署架构

```
前端(3000) → Web Server(Node.js) → vLLM(8003)     ← LLM 推理
                                → RAG API(8002)    ← Embedding + Rerank
                                → ChromaDB(8001)   ← 向量存储
```

**硬件**：NVIDIA A10 24GB，vLLM 占用 ~20GB，Rerank 模型 ~2.2GB

**全链路本地化**：Embedding（bge-m3）+ Rerank（bge-reranker-v2-m3）+ LLM（Qwen3-8B-FP8）全部跑在本地，无外部 API 依赖。

## 六、后续优化方向

1. **LLM Prompt 优化**：将候选的 keywords 传给 LLM，帮助区分相近职能
2. **知识库补充**：对匹配率低的领域（半导体、医疗）补充更细分的职能标签
3. **多 GPU 部署**：如有更多 GPU，可将 Rerank 模型搬到 GPU，Rerank 耗时从 2s 降至 ~200ms
4. **缓存机制**：对相同简历的职能分析结果做缓存，避免重复计算
